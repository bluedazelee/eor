## Context

EoR Tracker 是一個純前端 PWA（vanilla JS + HTML + CSS），無後端與外部依賴。現有的「確認本輪處理完畢」按鈕只在所有桌號皆為 `completed` 時啟用，無法應對賽事提前結束的例外情況。新按鈕需在不破壞現有完成流程的前提下加入。

## Goals / Non-Goals

**Goals:**
- 新增一個隨時可點擊的「強制結束本輪」按鈕
- 點擊後顯示含未完成桌數的二次確認對話框，防止誤觸
- 確認後執行與現有完成流程相同的清除與跳回起始畫面邏輯

**Non-Goals:**
- 不記錄或匯出強制結束時的未完成桌號清單
- 不修改現有「確認本輪處理完畢」按鈕的行為
- 不新增後端 API 或狀態同步機制

## Decisions

**決策 1：按鈕永遠可點擊，不依賴完成狀態**
- 選擇：按鈕始終 enabled，靠確認對話框防誤觸
- 棄選：用 disabled + tooltip 提示；但若沒有任何按鈕可按，緊急情況下使用者會卡住
- 理由：確認對話框已足夠作為防護，且實作最簡單

**決策 2：確認對話框顯示未完成桌數**
- 選擇：使用原生 `confirm()` 並在訊息中帶入未完成桌數
- 棄選：自訂 modal；對於 PWA offline 使用場景，原生 confirm 最可靠且無需額外 CSS/JS
- 理由：與現有 `btnFinishRound` 的確認邏輯保持一致，降低維護成本

**決策 3：清除邏輯與現有完成按鈕共用**
- 選擇：抽出 `resetAndReturnToSetup(roundName)` 函式，兩個按鈕共用
- 棄選：複製貼上相同程式碼；重複邏輯難以維護
- 理由：DRY 原則，單一清除入口更安全

**決策 4：按鈕樣式使用警示色（橘/紅）**
- 選擇：獨立 CSS class（如 `btn-warning`），與成功色的 `btn-success` 區分
- 理由：視覺上提醒使用者這是危險操作，與完成流程區隔

## Risks / Trade-offs

- [誤觸風險] 按鈕永遠啟用，使用者可能在非必要情況下點擊 → 確認對話框顯示未完成桌數，文案明確標示為強制操作
- [UX 一致性] 兩個按鈕並排，功能相近但語意不同 → 透過顏色（成功色 vs 警示色）及按鈕文字清楚區分
