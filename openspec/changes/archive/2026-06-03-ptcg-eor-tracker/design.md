## Context

在寶可夢卡牌 (PTCG) 賽事中，裁判長與桌協需要在每輪時間結束時 (End of Round, EoR) 迅速追蹤未結束的桌次。這款工具需要能在網路不穩定的比賽場地中運行，並且提供快速的單手操作介面，以卡片形式即時登記與更新各桌的狀態。

## Goals / Non-Goals

**Goals:**
- 提供一個可在行動裝置與桌面瀏覽器完美安裝與離線運作的 PWA SPA 工具。
- 提供輸入輪次與桌號區間以生成桌號圖卡的功能。
- 支援「尚未結束」、「已分配」、「已完成」三種狀態的單擊循環切換。
- 當切換為「已分配」時，自動標記並顯示當前的系統時間。
- 支援透過 Checkbox 控制是否隱藏「已完成」的桌號。
- 支援 LocalStorage 狀態持久化，防止重新整理遺失紀錄。
- 提供進度條與明確的完成/未完桌數統計。

**Non-Goals:**
- 不與任何外部賽事管理系統 (如 TOM 或 Limitless) 進行即時 API 對接（所有資料皆在本地端運作）。
- 不提供裁判名單管理或複雜的多人即時協作功能。
- 不提供歷史輪次存檔的報表匯出功能。

## Decisions

### 1. 技術棧與框架選擇
- **決策：** 採用純原生技術 (Vanilla HTML5 / CSS3 / JavaScript)，不使用任何 React/Vue 框架或 TailwindCSS。
- **理據：** 原生技術的打包體積最小 (總大小小於 50KB)，對於 PWA 離線快取極為友善。載入與執行速度極快，能完全滿足這款輕量型工具的效能需求，並簡化 Service Worker 快取設定。

### 2. SPA 視圖切換機制
- **決策：** 採用 DOM 元素隱藏/顯示 (使用 `.hidden` 屬性) 的方式切換「開始/設定畫面」與「紀錄主畫面」。
- **理據：** 由於此工具只有兩個主要畫面，且不涉及複雜的網頁 URL 導航，直接使用 DOM 操作可以免去引入第三方 Routing 函式庫的複雜度，使程式碼保持簡潔與高可讀性。

### 3. PWA 快取策略
- **決策：** Service Worker 採用 **Cache-First (快取優先)** 策略快取靜態資源 (`index.html`, `style.css`, `app.js`, `manifest.json`, 圖示)，並在每次變更版本時更新快取版本號。
- **理據：** 這個工具是 100% 的本地端應用，沒有與伺服器進行資料傳輸。Cache-First 能確保即便在完全沒有網路訊號的體育館中，應用程式也能在 1 秒內迅速啟動。

### 4. 狀態管理與 LocalStorage 持久化
- **決策：** 使用一個全域變數儲存當前輪次資料、設定之桌號範圍，以及各桌號的狀態陣列。每次卡片被點擊切換狀態時，直接呼叫一次資料序列化函式，將整個 JSON 結構寫入 `localStorage.setItem('ptcg_eor_data', ...)`。
- **理據：** LocalStorage 讀寫速度對於 100 桌左右的資料量而言微乎其微。這樣做能最簡單、最保險地在瀏覽器意外重新整理或斷電時還原全部的追蹤狀態。

## Risks / Trade-offs

- **[Risk] 瀏覽器清除 LocalStorage 導致資料遺失**  
  *Mitigation:* 對於這種臨時性 EoR 追蹤工具，資料只需保存一輪或一天。在每次開啟新一輪時皆會要求使用者重新設定。同時，會在介面加註提示，若瀏覽器處於隱私瀏覽模式，資料可能會在關閉分頁後清除。
- **[Risk] 手機點擊誤觸**  
  *Mitigation:* 由於裁判在場中是站立且走動操作，卡片必須設計得足夠大（例如在手機版每行顯示 3 至 4 列，寬高適中），並且卡片間距足夠，以減少誤觸機率。狀態顏色需有極高對比（紅、黃、綠/灰）。
