## Context

目前的 PTCG EoR Tracker 管理 200+ 桌時，裁判需逐張卡片點擊才能標記狀態。實際使用情境是語音回報，模式固定為兩種：「X到Y桌，除了 A,B,C，其餘皆完成」或「A,B,C 已完成」。現有的「全部設為已完成」僅覆蓋極端情況（全數完成），無法處理部分批量操作。

架構限制：純 Vanilla JS + CSS，無框架，目標裝置為手機，單手拇指操作。

## Goals / Non-Goals

**Goals:**
- 提供符合兩種語音回報語意的批量操作 UI
- 以底部彈窗（Bottom Sheet）保留足夠的輸入空間
- 支援範圍排除與指定桌號兩種輸入模式
- 即時預覽將影響的桌次數量
- 整批操作為單一 undo 步驟
- 提供搜尋列，輸入桌號後捲動至目標卡片並高亮

**Non-Goals:**
- 批量標記為非「已完成」的狀態（已有 undo 可撤銷誤操作）
- 視覺框選模式（tap 多張卡片）：手機上逐張點擊不比文字輸入快
- 部分搜尋（輸入 "4" 顯示所有含 4 的桌號）：語音回報場景下桌號精確已知

## Decisions

### 1. 底部彈窗（Bottom Sheet）而非內嵌表單

控制列空間有限，且需要數個輸入欄位（範圍起始、範圍結束、排除清單）。底部彈窗從畫面底部滑出，符合手機拇指操作習慣，並能提供足夠的輸入區域。

*替代方案考慮*：在控制列展開一個 inline panel → 空間不足，且遮蓋卡片格線。

### 2. Tab 切換兩種模式，而非單一智慧解析欄位

兩種回報語意結構差異明顯：第一種有「範圍」概念，第二種是純清單。Tab 讓使用者在聽到回報的第一句話時立刻選擇對應模式，避免解析歧義。

*替代方案考慮*：單一欄位自動偵測 → 在手機鍵盤上難以理解輸入規則，容易出錯。

### 3. 空格與逗號均視為分隔符

手機數字鍵盤打空格比切換到符號鍵盤打逗號快，兩者同時支援可減少使用者心智負擔。

### 4. 即時預覽桌數（不需額外確認對話框）

彈窗內的「確認執行」按鈕即是確認動作，加上即時預覽（「將標記 X 桌為已完成」）提供操作前的視覺確認。EoR 時間緊迫，額外的 confirm dialog 增加摩擦。undo 已提供足夠的撤銷保障。

*替代方案考慮*：保留 confirm dialog → 與現有 mark-all-completed 一致，但增加不必要步驟。

### 5. 單次 commitState() 覆蓋整批操作

批量標記在語意上是一次操作，使用者按下「確認執行」前已透過預覽確認。整批執行前呼叫一次 `commitState()`，undo 一步還原全部變更，不消耗多個 undo slot。

### 6. 搜尋採高亮捲動，不篩選隱藏其他卡片

篩選隱藏會讓使用者失去對相鄰桌號的上下文，且需要清空搜尋才能恢復正常視圖。高亮捲動的侵入性低，適合「快速確認某桌狀態」的使用場景。

### 7. 搜尋為精確比對（整數完全符合）

語音回報場景下桌號精確已知，不需要模糊比對。

## Risks / Trade-offs

- **輸入錯誤的非數字字元** → 解析時忽略非數字 token，預覽數字自動更新，使用者可即時發現異常
- **排除桌號超出範圍** → 靜默忽略，不影響結果，預覽已顯示最終數量
- **搜尋桌號不在當前範圍** → 搜尋列下方顯示「找不到桌號 X」提示，輸入框以紅色邊框標示
- **批量操作時有篩選器啟用（隱藏已完成 / 只顯示加時）** → 批量操作作用於 `state.tables` 全集，與畫面顯示篩選狀態無關
